# Backtesting And Score Optimization

Alpha 的回测目标是验证评分算法是否在历史区间中产生可复核的超额收益信号，而不是承诺未来收益。所有结论必须保留数据边界、失败清单和统计显著性。

## Core Rules

- **短结论优先**：先回答是否跑赢股票池、是否有统计显著性、覆盖率是否足够，再展开 TopN 和算法细节。
- **防未来函数**：评分时间点 `asOf` 只能使用 `asOf` 当日及以前的数据。`asOf` 之后到 `end` 的行情只能用于事后收益评估。
- **保留全量明细**：每次回测都要输出 `scored_*.csv`、`topN_*.csv`、`skipped_*.csv`、`kline_fetch_failures.csv`、`period_summary.csv`、`optimization_grid.csv`、`universe_field_audit.csv`、`backtest_summary.json` 和 HTML 报告。启用 walk-forward 动态选参时，`topN_*.csv` 是全局静态 selected-param 产物，不能用来复算 `walk_forward_summary.csv`；逐期权威明细必须使用 `walk_forward_topN_<asOf>_<end>.csv`、`walk_forward_scored_<asOf>_<end>.csv` 和 `walk_forward_skipped_<asOf>_<end>.csv`。2026-06-07 之前没有 `walk_forward_` 前缀的旧目录，只能作历史诊断，不能用于特征门控、权重复算或 selector 升级。
- **必须比较大盘**：默认输出映射大盘基准收益，并计算 TopN 等权收益、TopN 推荐仓位加权收益、相对股票池超额、相对大盘超额。不能只拿股票池平均做唯一对照。
- **必须横向比较主流指数**：除映射基准外，HTML 和 CSV 必须单独展示上证指数、深证成指、沪深300、中证500、中证1000、创业板指、科创50、北证50、恒生指数、恒生国企、恒生科技等主流指数的区间收益和组合相对超额。
- **不能只用等权组合**：TopN 等权收益只是研究对照。正式汇报时优先看 alpha 推荐仓位加权收益；TopN 明细必须写出 `recommendedWeight`。
- **默认看净收益**：默认开启交易成本/滑点估算。调参目标、walk-forward、HTML 摘要和最终结论优先看扣成本后的净自适应收益、净推荐仓位收益和净超大盘；毛收益只作为对照。
- **默认检查可交易性**：默认跳过 asOf 当日停牌/零成交、近似封涨停且收在最高价的入场点；目标退出日若近似封跌停或零成交，会在允许窗口内顺延到下一可交易日。这是现实成交约束，不是用未来收益筛选。
- **默认检查远期收益可信度**：回测评估远期收益时，默认跳过明显不可信的复权/公司行为污染样本，例如 2 个月远期收益超过 500%、远期亏损超过 90%，或出现大比例价格跳变且成交量反向缩放的疑似合股/拆股口径断裂。`skipped_*.csv` 必须记录 `extreme_forward_return`、`extreme_forward_loss` 或 `suspect_corporate_action_jump` 以及进出场价、原始远期收益和触发阈值。
- **默认保留完整候选池，动态过滤只作显式敏感性工具**：动态股票池过滤只能使用 `asOf` 前已知的 20 日均成交额、价格、波动、可选 ST 标记，以及显式开启的组内 20/60 日趋势、相对强弱和上涨宽度。开启后必须和未开启版本对照。若 walk-forward 子策略组合没有提升，或过滤后覆盖不足/依赖当前概念快照，不得把动态过滤设为默认。
- **允许自适应指数袖珍仓位**：当 `asOf` 前对应指数 20/60 日趋势很强、而 TopN 相对指数优势不足时，可以在模拟组合收益中加入映射指数袖珍仓位，对比纯 TopN 与 TopN+指数的收益。这个仓位只能由 `asOf` 前数据决定，不能用未来区间收益倒推。
- **默认输出 walk-forward 子策略组合**：多区间回测必须输出 `walk_forward_ensemble_summary.csv`。默认 `TopK=1`，即用之前区间训练稳健分选择单个最稳参数 sleeve，优先争取超额收益；`TopK=3/5`、score 加权、risk 风险预算只作为分散敏感性对照。若多策略组合稀释净收益或净超大盘，不得设为默认。
- **TopN 也必须可验证**：默认固定 `--top 10`，但可以用 `--top-values` 把持仓数量克隆为多个候选 sleeve，让 walk-forward 只用历史窗口选择当期 TopN。若动态 TopN 没有同时提升平均净收益、净超大盘、最近窗口和最差窗口，不得设为默认。
- **候选必须和固定基线成对比较**：pool selector 或新策略候选不能只看自身平均收益。默认用 `--pool-selector-baseline <name>` 输出 `baselineComparison` 和 `trainedBaselineComparison`，按相同 period 成对计算平均收益差、胜率、最差差值、最近差值和近似 paired t-test。若平均增量很小、胜率弱、最差差值恶化或近似 p 值没有支持，不得升级默认。这个门禁用于降低多次试验/网格搜索带来的过拟合；小样本 p 值只是反证工具，不是未来收益保证。
- **不要美化失败**：如果股票池覆盖不完整，必须说明输入数、成功评分数、跳过数、失败原因分布。不能把部分样本结果说成全市场结论。
- **显著性要求**：默认使用 TopN 与非 Top 样本的 Welch t-test。Top10 样本小，p-value 只能作为信号研究证据；`p < 0.05` 才能说“达到 5% 显著性”。
- **避免过拟合**：只有一个区间时，参数网格比较只能叫“区间内比较”，不能叫样本外优化。多区间时默认最后一个区间做 holdout，前面区间训练排序。排序目标必须同时看自适应组合绝对收益和相对大盘超额，不能只用超额收益压掉更赚钱、且仍有正超额的参数。
- **必须看 walk-forward**：多区间回测除了最后 holdout，还要输出 walk-forward 结果：每个测试区间只能用它之前的区间选择参数，再评估当前区间表现。walk-forward 比单次最后留出更接近真实迭代。
- **控制选参噪声**：当参数池很大、训练窗口很少时，walk-forward 容易追逐小样本 winner。可以显式设置生产 incumbent、切换门槛、incumbent policy 和探索参数 warm-up；候选参数必须在历史训练稳健分上有足够优势才允许替换 incumbent。`rolling` policy 会让切换后的 sleeve 成为下一期 incumbent，适合模拟真实线上策略迁移；`stable` policy 每期都重新和生产基线比较，适合复核实验参数是否真的持续打败当前默认。该机制仍只能使用当前测试区间之前的数据。
- **标注股票池偏差**：如果使用当前构建的 universe 回测更早日期，必须说明幸存者偏差和概念成分时间穿越风险。严格版本需要历史 universe 快照。

## Offline Data Mode

本仓库只保存脚本和测试，不保存离线行情数据。训练数据默认放在 `$ALPHA_DATA_HOME/cache/offline_datasets`，未设置 `ALPHA_DATA_HOME` 时等价于 `~/.codex/alpha/cache/offline_datasets`。

`--offline-data` 会优先读取本地离线训练集；A 股走 Qlib daily bin，港股通走 completed-only CSV。缺失时仍可按缓存或 provider 回退。

```bash
node scripts/alpha_backtest.js \
  --universe $ALPHA_DATA_HOME/output/current-full-alpha-v5-20260608-final/current_full_universe.csv \
  --period 2026-04-01:2026-06-01 \
  --top 10 \
  --lookback 420 \
  --min-history 65 \
  --offline-data \
  --out-dir $ALPHA_DATA_HOME/output/backtest-offline-example
```

`--offline-only` 用于验证或训练时的本地闭环：它等价于 `--offline-data --cache-only`，不会联网补数。股票 K 线必须来自 Qlib 或港股通 CSV；指数对比若本地没有缓存，会记录到 `benchmark_fetch_failures.csv`，HTML 中对应指数收益保持缺失，不得当成 0。

```bash
node scripts/alpha_backtest.js \
  --universe $ALPHA_DATA_HOME/output/current-full-alpha-v5-20260608-final/current_full_universe.csv \
  --period 2026-04-01:2026-06-01 \
  --top 10 \
  --lookback 420 \
  --min-history 65 \
  --offline-only \
  --out-dir $ALPHA_DATA_HOME/output/backtest-offline-only-example
```

## Pool Selector Command

当需要在多个已完成的 walk-forward 股票池结果之间做二级动态选池时，用 `--pool-selector-dirs` 读取各目录下的 `walk_forward_summary.csv`，只根据之前窗口的池级表现选择当前窗口，并输出 `walk_forward_pool_selector_summary.csv` 和 `pool_selector_summary.json`。该模式会直接退出，不重新抓取 K 线或重跑单票评分。

```bash
node scripts/alpha_backtest.js \

  --pool-selector-min-train 6 \
  --pool-selector-initial merged \
  --pool-selector-baseline merged \
  --pool-selector-score-excess-weight 0.35 \
  --pool-selector-include-warmup \
  --out-dir $ALPHA_DATA_HOME/output/backtest-pool-selector-cli-v1-diagnostic-17p-cacheonly-lookback520-20260607
```

## PIT Cache Universe

当需要降低“用当前股票池倒推历史”的未来成员泄漏时，用 `build_pit_cache_universe.js` 从本地腾讯 K 线缓存按每个 `asOf` 生成近似 point-in-time 股票池。该脚本只要求股票在 `asOf` 当日及之前有足够有效交易历史、最近交易日不超过 stale 阈值、可选满足 20 日均成交额；它不要求 `asOf` 之后有未来行情，因此不会用未来收益或未来覆盖率决定当期成员。

```bash
node scripts/build_pit_cache_universe.js \
  --cache-dir $ALPHA_DATA_HOME/cache/kline \
  --as-of-dates 2024-12-02,2025-01-02,2025-02-05,2025-03-03,2025-04-02,2025-05-06,2025-06-03,2025-07-01,2025-08-01,2025-09-01,2025-10-09,2025-11-03,2025-12-01,2026-01-02,2026-02-02,2026-03-02,2026-04-01 \
  --out-dir $ALPHA_DATA_HOME/output/pit-cache-universe-v1-20260607 \
  --lookback 520 \
  --min-history 65 \
  --max-stale-calendar-days 14
```

`alpha_backtest.js` 支持用 `--period-universe-dir <dir>` 接入这类输出：每个 period 会读取 `pit_universe_<asOf>.csv`，K 线加载也会按单票实际出现的最早 asOf / 最晚 end 校验窗口，避免后来才入池的股票被全局最早 asOf 误剔除。

```bash
node scripts/alpha_backtest.js \
  --period-universe-dir $ALPHA_DATA_HOME/output/pit-cache-universe-v1-20260607 \
  --periods 2024-12-02:2025-02-05,2025-01-02:2025-03-03,2025-02-05:2025-04-02,2025-03-03:2025-05-06,2025-04-02:2025-06-03,2025-05-06:2025-07-01,2025-06-03:2025-08-01,2025-07-01:2025-09-01,2025-08-01:2025-10-09,2025-09-01:2025-11-03,2025-10-09:2025-12-01,2025-11-03:2026-01-02,2025-12-01:2026-02-02,2026-01-02:2026-03-02,2026-02-02:2026-04-01,2026-03-02:2026-05-04,2026-04-01:2026-06-01 \
  --top 10 \
  --lookback 520 \
  --min-history 65 \
  --provider tencent \
  --cache-only
```

边界必须写清楚：PIT cache universe 仍只是本地缓存近似。它能减少未来成员泄漏和短历史新股穿越，但不能补回退市股票，也没有历史概念、行业、财报或公告快照。

## Tencent Kline Backfill

当 520 日缓存不足以支持更早 `asOf` 时，用 `backfill_tencent_kline_cache.js` 把已有腾讯 K 线缓存从 520 日补到 900 日。该脚本会发现已有 `${symbol}_520_qfq.json`，重新请求 `${symbol}_900_qfq.json`，并写出 `kline_backfill_900_manifest.json`；成交量单位沿用市场归一化规则：深市/北交所按“手”乘 100，沪市/科创和港股按“股”处理。

```bash
node scripts/backfill_tencent_kline_cache.js \
  --cache-dir $ALPHA_DATA_HOME/cache/kline \
  --source-lookback 520 \
  --target-lookback 900 \
  --concurrency 24
```

2026-06-08 实跑结果：1710 个已有 520 日缓存标的中，1690 个补成 900 日，20 个已存在跳过，失败 0。随后用 900 日缓存生成 `$ALPHA_DATA_HOME/output/pit-cache-universe-21p-900-v1-20260608`，实际包含 25 个 asOf，覆盖从 `2024-04-01` 的 1548 只逐步升到 `2026-04-01` 的 1624 只；拆分目录在 `$ALPHA_DATA_HOME/output/pit-cache-universe-25p-900-variants-v1-20260608`。全量 25 期长样本回测可能超过 Node 默认 4GB heap，实跑全量 PIT 时需要 `node --max-old-space-size=8192`，否则会在评分排序阶段 OOM。

## Period Universe Variants

当需要把一组按 `asOf` 生成的 period universe 继续按市场结构拆分时，用 `build_period_universe_variants.js`。它会读取 `pit_universe_<asOf>.csv`，在每个 asOf 内分别输出 A股、A股剔北交所、深市、沪市、港股通五个市场目录，并额外输出深主板、创业板、沪主板、科创板、北交所等交易板目录；每个目录仍保留原始 `pit_universe_<asOf>.csv` 文件名，可直接继续传给 `alpha_backtest.js --period-universe-dir`。

```bash
node scripts/build_period_universe_variants.js \
  --input-dir $ALPHA_DATA_HOME/output/pit-cache-universe-v1-20260607 \
  --out-root $ALPHA_DATA_HOME/output/pit-cache-universe-variants-v1-20260607
```

市场目录只使用每期 CSV 已有的 `market` 字段做成员拆分；交易板目录只使用 `market` 与 asOf 文件里已有的代码前缀，例如创业板 `300/301`、科创板 `688/689`。脚本不使用未来收益。若 A股与 A股剔北交所 code 集合一致，收益理论上相同；但若 theme/source/reason 等静态字段进入评分，仍需保留敏感性说明。

2026-06-07 复核发现一个重要回测边界：此前 walk-forward 用“前一个滚动窗口”训练，但区间是月度滚动、持有期约两个月，因此相邻历史窗口的 `end` 常晚于当前 `asOf`。例如在 `2026-04-01` 做选择时，`2026-03-02:2026-05-04` 的收益尚不可知；若直接按索引回看，会把尚未结束的重叠窗口收益纳入训练。旧的 18%/19% 结果应标记为 overlapping-window diagnostic，不再作为严格样本外 production 证据。

- `--ensemble-top-k <n>`：walk-forward 子策略组合 TopK，默认 1。提高 TopK 会更分散但可能稀释强信号，降低 TopK 会更进攻但更容易受单参数过拟合影响。
- `--ensemble-weighting equal|score|risk`：walk-forward 子策略组合权重方式，默认 `equal`。`score` 使用训练稳健分 softmax 加权；`risk` 使用训练期收益、超额、命中率、波动、下行波动和最差窗口做风险预算。两者都适合敏感性对照；若未在多窗口净收益和净超大盘上超过 `equal`，不得设为默认。
- `--ensemble-score-temperature <n>`：`score` 权重的 softmax 温度，默认 20。温度越高越接近 winner-take-all，越低越接近等权。
- `--wf-stable-param <name>`：设置 walk-forward 的生产 incumbent。每个测试窗口先用历史训练窗口打分；若 challenger 的训练稳健分没有超过 incumbent 足够多，则保留 incumbent，避免小样本频繁切换。
- `--wf-switch-margin <n>`：设置 challenger 替换 incumbent 所需的训练稳健分优势，默认 0。门槛不是越高越好；必须和 `--wf-incumbent-policy` 一起审计，因为 rolling policy 下高门槛可能让已切换的实验 sleeve 变得更粘。
- `--wf-incumbent-policy rolling|stable`：设置 walk-forward 的 incumbent 口径，默认 `rolling`，即切换后的 sleeve 成为下一期 incumbent。`stable` 会在每个测试窗口重新拿 challenger 和 `--wf-stable-param` 比较，适合验证实验参数相对生产基线的持续优势。2026-06-07 的 v53/v54 复核显示，合并宽池最佳为 `--wf-stable-param balanced_reversal_stability_v51 --wf-switch-margin 0.01 --wf-incumbent-policy stable`。
- `--wf-current-gate off|fresh-v1|regime-v1|...|regime-v32`：设置 walk-forward 当前篮子质量门禁，默认 `off`。`fresh-v1` 只在训练分已经触发 fresh/rank_fresh challenger 切换时生效，并且只读取当期 `asOf` TopN 已生成的 `freshTrendScore`、`r20`、`relativeMomentumScore` 和 `industryResidualMomentumScore` 均值差，不读取未来收益。`regime-v1` 继承 `fresh-v1`，再追加当前过热降速 overlay：若当期原本选择 `balanced_reversal_stability_v51`，且 `balanced_reversal_stability_selective_exhaustion_cash_v56` 的 `exhaustionCashWeight` 已由 asOf 前指数 20/60 日降速、TopN 60 日涨幅、波动和相对确认不足触发，则仅替换当期 sleeve，不改变后续 incumbent 训练路径。`regime-v2` 改用 `balanced_reversal_stability_mature_exhaustion_cash_v57` 作为 overlay-only sleeve：它不参与普通训练切换，只在当期 `exhaustionCashWeight` 已由 asOf 前指数过热降速或 TopN 自身成熟长趋势、高波动、fresh 降温触发时替换当期结果。`regime-v3` 在 v57 现金 overlay 外追加 v58 beta recovery overlay：上一期已落地选择跑输映射大盘超过 5pct，且当前 v51 Top10 的 `benchmarkR20/R60` 仍低、`relativeR20` 和 `relativeMomentumScore` 较强时，用 v58 给指数袖珍仓位做恢复修复。`regime-v4` 把 v58 替换为 v59 stronger beta recovery，仍是 overlay-only，只调指数袖珍仓位强度，不改选股排序。`regime-v5` 把成熟动量 overlay 从 v57 换成 v61：当 TopN 自身成熟降温但基准 20/60 日趋势温和不拥挤时，把原本的成熟动量现金转为指数袖珍仓位，同时继续沿用 v59 recovery overlay。`regime-v6` 在 v5 基础上追加 v62 benchmark pullback catch-up：当 Top10 自身趋势成熟、相对大盘仍强、基准 20/60 日温和但 5 日刚回踩时，加入小指数袖珍仓位。`regime-v7` 在 v6 基础上把成熟动量温和基准轮动从 v61 换成 v63，仍只调指数袖珍仓位强度，不改选股排序。`regime-v8` 在 v7 基础上允许 `rank_benchmark_state_attack_v32` 作为强 beta 进攻 challenger，并在 v32 轻度跑输后允许 v59 beta recovery 用自身 asOf 状态接管当期，避免进攻 sleeve 阻断恢复 overlay。`regime-v9` 在 v8 基础上给 v32 强 beta 进攻增加 asOf 行业残差质量门禁：当 v32 Top10 的对应基准 20 日收益已经转正，但相对 incumbent 的行业残差动量优势深度为负时，先阻断 v32。`regime-v10` 在 v9 基础上把 beta recovery overlay 的检查放到 v32 进攻质量阻断之前，避免质量门禁误挡已满足恢复条件的 v59。`regime-v11` 在 v10 基础上增加 high-fresh rotation overlay：当 v51 的成熟动量 overlay 将要接管、但 v54 Top10 在 fresh、20 日趋势和相对动量上明显更强，且基准 20/60 日处于高位、行业残差优势明显为负时，优先让 v54 接管，避免成熟旧龙头在过热回落期继续拖累。`regime-v12` 继承 v11，并且只在 v11 fresh rotation 已触发后，把 v64 当 overlay-only lottery guard：若 v64 Top10 的 `lotterySpikeScore` 均值不低于 98，且 `freshTrendScore` 均值处在 70-82 的适中 fresh 区间，则用 v64 替换 v54；v64 不参与普通训练竞选。
- 2026-06-07 起，CLI 白名单还支持 `regime-v13` 到 `regime-v32`。`regime-v13` 继承 v12，并在中期基准仍上行、短期指数回踩、v32 Top10 相对强度足够时用 `rank_benchmark_state_attack_v32` 接管；`regime-v14` 继承 v13，并在 v54 的 fresh/20 日优势过强但相对动量和行业残差确认不足时回落到 `balanced_reversal_stability_mature_beta_rotation_stronger_v63`；`regime-v15` 继承 v14，并在指数不过热、中期仍上行、v54 中等 fresh 且行业残差很强时允许 fresh continuation 接管；`regime-v16` 继承 v15，并在指数主升、v32 广谱相对强、fresh 不过热、lottery 不高且 `industryResidualRankScore` 不拥挤时允许 v32 broad beta attack 接管；`regime-v17` 继承 v16，并在 v54 fresh 接管但 v32 早期广谱 beta 更强且不过热时允许 v32 接管；`regime-v18` 允许成熟/fresh 高位降速且波动抬升时切到 crash-cash overlay，但当前未升级；`regime-v19` 在 v62 pullback catch-up 已触发且 v59 指数袖珍仓位明显更强时允许 beta sleeve 升级；`regime-v20` 在 v59 beta recovery 已触发且 v67 高置信指数仓位明显更强时允许进一步提高 beta sleeve；`regime-v21` 在训练已切到 v54、但 v64 能用 asOf 特征显著压降 lottery/MAX 风险且不牺牲中等 fresh 时，允许 v64 直接接管；`regime-v22` 在训练已切到高 fresh v54、但行业残差 rank 确认偏弱，且 v52 同时具备低波动、稳定短反、换手稳定和相对动量确认时，允许 v52 接管；`regime-v23` 在训练直接切到 v54、但 v54 当期 fresh/量能确认偏弱，而 v28 具备中等 fresh、量能动量、相对强度且 lottery 风险不过高时，允许 v28 接管；`regime-v24` 在 v51 已被保留、但 v52 当期具备中等 fresh、相对强、行业残差 rank 强、换手稳定且 lottery 不高时，允许 v52 接管；`regime-v25` 在 v32 已被选中且指数中期仍上行、短期回踩、v28 具备强相对和中等量能确认时，允许 v28 接管；`regime-v26` 在 v64 已由 fresh rotation lottery guard 触发、但 v64 高 fresh 高 lottery 且相对动量/换手稳定不足，同时 v52 中等 fresh、稳定短反和换手稳定更好时，允许 v52 接管；`regime-v27` 在 v14 fresh 过度延展回落到 v63/v63_no_static 后，且 v52 处于弱指数 60 日背景、低波动、低尖峰、稳定短反和换手稳定时，允许 v52 接管；`regime-v28` 在 v32 回踩进攻已触发后，且 v28 同时具备高行业残差确认、量能动量、低 lottery 和入场不过热时，允许 v28 接管；`regime-v29` 在宽池早期修复、指数 20 日温和反弹但 60 日仍未修复、v51 被保留且 v52 具备稳定短反/行业残差 rank/低追涨约束时，允许 v52 作为 overlay-only 接管；`regime-v30` 在宽池深回踩、v32 回踩进攻已触发、且 v52 具备低 fresh 稳定短反、行业残差 rank、breadth 和入场不过热约束时，允许 v52 作为 overlay-only 接管；`regime-v31` 在宽池早期强量能、指数 20 日温和上行但 60 日未过热、v28 高相对强度/高 breadth/低 lottery 且入场不过热时，允许 v28 作为 overlay-only 接管；`regime-v32` 在 v51 已被保留、窄池早期强势但静态主题字段可能拖累时，允许 `balanced_reversal_stability_mature_beta_rotation_stronger_v63_no_static_theme` 作为 overlay-only 接管。当前最高验证口径见上方 Current Validated Candidate。
- `--wf-min-exploratory-periods <n>`：探索参数 warm-up。默认关闭；设置后，`rank_` 等复杂探索 sleeve 至少积累 n 个历史训练区间后才参与竞选，避免训练窗口太少时由复杂参数接管组合。
- `--include-experimental-params`：把实验参数组纳入优化竞选，默认关闭。实验参数会继续输出诊断字段，但不能在未验证前污染默认 TopK walk-forward。
- `--param-names a,b,c`：只使用指定参数组，自动允许选择 experimental 参数；用于隔离测试单个因子或少数候选，不需要把全部实验池一起纳入。
- `--no-static-theme`：关闭 universe CSV 里的静态主题/行业信号，只保留 K 线、指数相对强弱、流动性、波动、可交易性等 asOf 可控输入。用于检查 `concepts`、`relevance`、`source`、`industry` 等当前快照字段是否污染收益。
- `--include-no-static-params`：在参数池里追加每个参数的 `_no_static_theme` 克隆，让 walk-forward 用之前区间自行选择静态主题版或 no-static 版。若组合池没有同时改善平均收益、净超大盘、最近窗口和最差窗口，不得默认启用。当前最高候选只允许额外加入 `balanced_reversal_stability_mature_beta_rotation_stronger_v63_no_static_theme`，全量 no-static 克隆在合并宽池上明显退化，不得默认放开。
- `--include-no-static-param-names a,b`：只给指定源参数追加 `_no_static_theme` 克隆。适合当前 `v17 + v63_no_static_only` 这种窄开实验，可避免 `--include-no-static-params` 把所有 no-static 克隆放进候选池造成选参噪声。若同时使用 `--param-names`，传源参数名即可保留对应 no-static 克隆；未知源参数必须报错，不允许静默忽略。

## Current Scoring Inputs

当前 `backtest_engine` 使用这些 point-in-time 可控输入：

- 5/20/60 日收益趋势。
- 20 日趋势相对 60 日均速的加速项，用于识别刚启动而不是已经涨很久的标的。
- fresh-trend 新鲜趋势项：只用 `asOf` 前 `r5`、`r20`、`r60` 和 `acceleration20vs60` 计算 `freshTrendScore`、`freshAccelerationScore`、`trendMaturityPenaltyScore`，奖励 20 日新近加速、5 日不过热且 60 日没有明显老化的趋势，惩罚 60 日涨幅过高后动能衰减的 stale winner。TopN 明细输出 `freshTrendRankScore`，用于区分“刚启动的趋势”和“涨久后回撤概率变高的趋势”。
- 突破回踩承接项：只用 `asOf` 前 K 线计算 20 日高位回撤深度、5 日相对前 15 日成交额比例、是否守住 20 日均线、短期是否不过热，用于识别强趋势后的温和缩量回踩，而不是追直线过热票。TopN 明细输出 `pullbackAccumulationScore`、`pullbackDrawdown20`、`pullbackSupportRatio20`、`pullbackVolumeRatio5v20` 和 `pullbackRankScore`。
- 量价确认项：只用 `asOf` 前 K 线计算 5 日成交额相对前 15 日扩张、短期涨幅是否不过热、20 日趋势和加速是否配合，用于验证动量是否有成交确认。TopN 明细输出 `volumeMomentumScore`、`volumeTurnoverRatio5v20` 和 `volumeMomentumRankScore`。该思路来自价格动量与交易量联合预测的业界/学术经验，但必须通过本地 walk-forward；不能因为理论合理就升级默认。
- 52 周高点邻近项：只用 `asOf` 前最近 252 个交易日的历史最高价、当前价距离最高价比例、高点距今天数和 20/60 日趋势计算长期动量邻近信号。TopN 明细输出 `high52wScore`、`high52wDistance`、`high52wDaysSinceHigh` 和 `high52wRankScore`。该思路可作为突破/动量诊断，但当前只在实验参数中使用，不能因为学术上常见就默认化。
- 60 日回撤。
- 20 日实现波动率。
- 20 日平均成交额或成交量估算成交额。
- 腾讯日线成交量必须按市场单位归一化：沪市/科创和港股原始成交量按“股”处理，深市/北交所原始成交量按“手”乘 100。2026-06-07 已加入旧版沪市腾讯缓存的只读归一化，避免把沪市成交额放大 100 倍后污染流动性 rank、交易成本和股票池过滤。
- 对应市场基准指数在 `asOf` 前的 5/20/60 日收益，以及股票相对该指数的 5/20/60 日超额动量。TopN 明细输出 `benchmarkTrendRankScore` 和 `relativeMomentumRankScore`，用于诊断指数 beta 状态和相对强弱是否主导选股。
- 同行业样本在 `asOf` 前的平均 20/60 日趋势、相对基准趋势和 20 日上涨宽度，用于识别赛道内扩散。
- 行业内残差动量：只用 `asOf` 前同一行业样本均值计算 `industryResidualR20/R60` 和 `industryResidualRelativeR20/R60`，用于识别“行业整体一般、但个股已经跑赢同行”的候选。TopN 明细输出 `industryResidualMomentumScore` 和 `industryResidualRankScore`。该思路来自行业动量/行业内相对特征研究，但当前只作为实验因子，必须用本地 walk-forward 验证。
- TopN 对应基准指数在 `asOf` 前的强 beta 状态，用于决定模拟组合里是否加入指数袖珍仓位。
- 实验性市场状态仓位：`benchmarkOverlay` 只用 `asOf` 前基准 20/60 日趋势和 TopN 相对强弱决定指数袖珍仓位；`defensiveCash` 只用 `asOf` 前基准 20/60 日弱势决定防守现金权重；`exhaustionCash` 只用 `asOf` 前指数 20/60 日降速、TopN 60 日涨幅、波动率、20 日相对收益和相对动量分判断是否降低过热后动能衰减风险；`maturityBenchmarkOverlay` 只在 TopN 自身成熟降温且基准 20/60 日趋势温和不拥挤时，把本应进入现金的成熟动量仓位转为指数袖珍仓位；`recoveryOverlay` 只用上一期已发生的净超大盘结果和当前 asOf TopN 的指数/相对强弱状态决定是否临时提高指数袖珍仓位。这些仓位规则都不得读取未来区间收益。
- 静态主题相关度、概念数量和人工补充标记。
- `universe_field_audit.csv` 会把股票池字段分成 `identifier`、`current_market_snapshot`、`current_valuation_snapshot`、`current_theme_snapshot`、`current_industry_snapshot` 等类型，并标注是否被评分使用。当前 `price`、`pct_chg`、`amount`、`pe`、`pb`、`total_mv` 只作为上下文保留，未进评分；`concepts`、`relevance`、`source`、`industry` 会影响主题分、行业动量或来源组强度，因此在没有历史 point-in-time 快照前必须用 `--no-static-theme` 做敏感性对照。
- 估算交易成本：股票端按基础 round-trip 成本、20 日流动性压力和 20 日波动率冲击扣减；指数袖珍仓位用更低基础成本扣减。TopN 明细输出 `executionCostRate`、`netForwardReturn`、`netBenchmarkForwardReturn`。
- 成本感知推荐仓位：`recommendedWeight` 会对高交易成本个股降权，避免低流动性或高波动票在组合里过度集中。
- 入场/退出可交易性：用 `asOf` 当天 K 线和前一交易日收盘价判断停牌/零成交、成交额下限、A 股主板/创业板/科创板/北交所近似涨停封板；目标退出日若近似跌停封板，则最多顺延 `maxExitDelayDays` 个交易日。`scored_*.csv` 输出 `entryDayReturn`、`entryTurnover`、`entryCloseAtHigh`、`targetExitDate`、`exitDate`、`exitDelayDays`、`exitReason`；`skipped_*.csv` 记录 `entry_limit_up_like`、`exit_limit_down_like` 等原因。
- 可选动态股票池过滤：显式开启后，用 `scoreAtDate` 已生成的 `avgTurnover20`、`priceAsOf`、`vol20` 和名称 ST 标记过滤当期可评分样本；`skipped_*.csv` 会记录 `universe_low_avg_turnover`、`universe_low_price`、`universe_high_volatility`、`universe_st_stock` 等原因。
- 可选动态行业/主题/来源强度过滤：显式开启后，先在当期可评分样本上计算 `dynamicGroupR20/R60`、`dynamicGroupRelativeR20/R60`、`dynamicGroupBreadth20` 和 `dynamicGroupScore`，再过滤弱组。过滤只能使用 asOf 前价格/基准派生指标；行业、概念或来源归属仍来自 universe CSV。若过滤后低于 `minDynamicGroupRemaining` 或参数内 `minRemainingRatio` 对应的覆盖率底线，工具会 fail-open 回原池，避免空组合或过窄主题池。`scored_*.csv` 和 `skipped_*.csv` 会输出 `dynamicGroupKey`、`dynamicGroupCount`、`dynamicGroupScore`、`dynamicGroupBreadth20` 等诊断字段。
- 可选动态行业/主题/来源强度排序或微调分：参数内设置 `dynamicGroupMetrics.enabled` 后，可在不过滤股票池的情况下输出同样的动态组指标。rank 参数可用 `rankDynamicGroupWeight` 生成 `dynamicGroupRankScore`；原始分参数可用 `dynamicGroupMetrics.scoreWeight` 小权重混入 `score`，并在 CSV 输出 `scoreBeforeDynamicGroupBlend` 和 `dynamicGroupScoreBlendWeight`。该机制只解决“组强度如何进入评分”的工程问题；是否有效必须看 walk-forward，不得因为概念或来源池当期强就默认加分。
- 短期反转/换手稳定性：只用 `asOf` 前 K 线计算 3/5 日冷却、20 日最大短期回撤、20 日均线支撑比例、20 日换手/成交额稳定性和短期波动惩罚，用于识别“中期趋势仍在、短期不过热、回踩后仍有承接”的候选。TopN 明细输出 `shortTermReversalScore`、`shortTermReversalR3`、`shortTermDrawdown20`、`shortTermSupportRatio20`、`turnoverCv20`、`turnoverStabilityScore`、`shortTermReversalRankScore` 和 `turnoverStabilityRankScore`。该思路参考 A 股短期反转、低波动和成交稳定相关经验，但只作为本地 walk-forward 验证后的候选因子，不能因理论合理直接默认化。
- 远期收益可信度护栏：收益评估阶段默认启用 `returnSanity`，只影响历史回测/训练标签，不影响 live 打分。默认阈值为 `maxForwardReturn=5`、`minForwardReturn=-0.90`、疑似公司行为跳变 `corporateActionPriceRatio=4` 且成交量反向缩放阈值 `corporateActionInverseVolumeRatio=0.35`。异常样本会进入 `skipped_*.csv`，不会参与参数训练、TopN 排序或组合收益统计。

参数网格同时保留两类算法：

## Optimization Workflow

1. **Smoke test**：用 `--limit 30` 验证抓取、缓存、CSV、HTML 全链路。
2. **Full run**：取消 `--limit`，跑完整 universe；等待所有失败被记录。
3. **Read summary**：先看 `period_summary.csv` 的净自适应组合收益、净超大盘、胜率、p-value、覆盖率，同时用毛收益判断成本敏感性，并检查因 `universe_low_avg_turnover`、`entry_limit_up_like` 等原因跳过的样本数。
4. **Audit universe fields**：打开 `universe_field_audit.csv`，确认当前价格、估值、概念、行业等快照字段是否进入评分。若评分使用了 `current_theme_snapshot` 或 `current_industry_snapshot` 字段，必须跑 `--no-static-theme` 或 `--include-no-static-params` 对照，不能直接把静态快照收益当作真实 alpha。
5. **Check holdout**：多区间时重点看最后一个区间。训练区间好、留出区间差，说明过拟合；但如果参数仍有正绝对收益且长期平均显著更强，不要只因单一 holdout 超额略差就丢弃，必须放到 walk-forward 里再验证。
6. **Check walk-forward**：打开 `walk_forward_summary.csv`，看滚动选参的训练稳健分、训练净收益、训练净超额、平均净自适应收益、平均净超大盘、命中率和最后几个窗口是否退化。若 walk-forward 与静态 holdout 冲突，优先把冲突当成不稳定信号，而不是直接挑更好看的数字。
7. **Check selection noise controls**：如果候选池超过 10 个 sleeve，必须对照无控制、`--wf-stable-param balanced_v2 --wf-switch-margin 0.05` 和候选 incumbent 口径。查看 `candidateParam`、`incumbentParam`、`selectionReason`、`candidateScoreAdvantage` 和 `filteredParamCount`，确认切换是由稳定训练证据触发，而不是小样本噪声。

## Reporting Template

回测完成后按这个顺序回复：

1. **结论**：自适应组合收益、指数袖珍仓位、纯 TopN 推荐仓位收益、TopN 等权收益、股票池平均、映射大盘基准、相对大盘超额、主流指数横向排名、p-value、是否显著。
2. **覆盖率**：输入数、成功评分数、失败/跳过数、主要失败原因，单列入场不可交易原因。
